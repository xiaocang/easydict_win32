# Migration Experience

## Migration Completeness (do not re-investigate)

- The default rs build (no `retained-dotnet-workers` / `hybrid-dotnet-runtime-packaging` features, no `EASYDICT_RUNTIME_PROFILE=hybrid`) is already fully Rust-native: no `dotnet.exe` / hostfxr / CoreCLR / PowerShell / worker spawning, no `.NET` fallback, and the not-yet-native scenarios return a local "requires a Rust-native route" error instead of touching `.NET`. Verified 2026-06-16 by `Invoke-RsCoreSliceValidation.ps1 -Profile rust-only-boundary` (all 10 steps green, parallel UI isolated/restored). Re-confirm with that lane rather than re-grepping the whole tree.
- All remaining `.NET` is intentional coexistence the migration plan preserves, NOT a pending migration item: (1) the legacy `dotnet/` WinUI tree, (2) the `retained-dotnet-workers` compat workers (`compat_client.rs`, `compat_protocol.rs`, `bin/easydict_ipc_mock.rs`, ~82 `#[cfg(feature = "retained-dotnet-workers")]` guards), gated behind both the Cargo feature AND the `hybrid` runtime profile, and (3) the vendored `lib/PdfPig` .NET source tree (used by legacy dotnet + as a 638KB CJK font test fixture `PMingLiU.ttf` referenced from rs tests). Removing any of these is a destructive, plan-contradicting decision — surface it, do not do it unilaterally as a "migration slice".
- `translation_cache.rs::dotnet_language_name(...)` is a Rust function emitting `.NET`-compatible cache-key strings for cache parity — naming parity, not a runtime dependency. Do not "migrate" it.
- The genuine remaining work to literally remove ALL `.NET` is NOT small unported code: it is either dismantling the intentional coexistence layer above, or closing inherent model-capability gaps (LocalAI language pairs / LongDoc scenarios that no native WindowsAI/Foundry/OpenVINO path can serve). Treat the rs-version migration goal as achieved for the default build; further reduction needs an explicit scope decision.
- `lib/PdfPig` is an ACTIVE dependency of the legacy dotnet build — `dotnet/Easydict.Win32.sln` references 5 of its projects (UglyToad.PdfPig{,.Core,.Fonts,.Tokenization,.Tokens}). It is not removable while the legacy/hybrid coexistence build exists. The rs side does NOT use it at runtime (rs PDF = `lib/easydict-pdf-render` pdfium + `lib/easydict-pdf-overlay`); rs only reaches into its `UglyToad.PdfPig.Tests/Fonts/TrueType` directory for font fixtures (Roboto + PMingLiU) in `font_metrics_behavior.rs`, `resource_download_behavior.rs`, `long_document.rs`, `settings_status.rs`. Decoupling those fixtures enables no deletion and removes no runtime `.NET`, so it is not a migration slice — don't spend effort on it.
- A repo-wide search for pending Rust work (`TODO`/`FIXME`/`unimplemented!`/`todo!`/`暂时保留`/`尚未迁` in `rs/crates/*/src` + `lib/*/src`, excluding the vendored `lib/PdfPig` .NET tree) returns ZERO hits. There is no half-finished rs-native code awaiting completion; the runtime migration is implementation-complete.
- UPDATE 2026-06-16: the `retained-dotnet-workers` coexistence feature is being RETIRED (the plan's final target is Rust-only). The full removal is committed on branch `refactor-remove-retained-workers` (commit `c0d213c0`), not yet merged into `next/rs`. After merge, `retained-dotnet-workers`, `compat_client.rs`, `compat_protocol.rs`, `easydict_ipc_mock`, and the `retained-worker-ipc` validation profile no longer exist. Removing it is safe and behaviorally inert for the default build (the feature code never compiled there). Remaining .NET after merge: the legacy `dotnet/` tree and vendored `lib/PdfPig` only (both legacy-build-only).

## Retiring a Cross-Cutting Cargo Feature (retained-dotnet-workers)

- Removing a feature that the default build never compiled is mostly DELETION + un-gating, with zero default-build behavior change: delete every `#[cfg(feature = "X")]` item, and strip the cfg from every `#[cfg(not(feature = "X"))]` (reduce `#[cfg(all(windows, not(feature="X")))]` to `#[cfg(windows)]`). Watch for `#[cfg(feature)] fn foo` / `#[cfg(not(feature)] fn foo` PAIRS — delete the feature variant, un-gate the `not` one; otherwise un-gating creates a duplicate-definition error.
- Order top-down (kill the feature definition's effect first) so the default build stays green throughout; the feature build ceases to exist, which is the goal. `required-features = ["X"]` on a `[[bin]]` ERRORS if X is undefined, so delete that bin entry together with the feature. A bare `#[cfg(feature = "X")]` on an undefined feature is NOT an error (evaluates false), so leftover cfgs don't block compilation — but finish the cleanup so `grep` is clean.
- Run such a refactor in an isolated git worktree (Agent `isolation: "worktree"`) to avoid racing a parallel agent editing the main worktree. PITFALL: the auto-provisioned agent worktree may be based on a STALE commit (here `ab92efe8`, which predates `rs/`), and the target branch may already be checked out in a sibling worktree (git forbids two worktrees on one branch). The subagent must create its own branch off the correct base (`next/rs` HEAD). Also, a fresh worktree may have `lib/PdfPig` (a submodule) uninitialized — its font-fixture tests then fail environmentally; don't mistake that for a refactor regression.
- Merge coordination: a worktree-branch refactor cannot be merged into `next/rs` while the main worktree is dirty (parallel UI work) — git blocks the merge. Commit the refactor to its feature branch to make it durable, then merge once the main tree checkpoints. If the refactor and a prior uncommitted main-tree edit both touched the same file (e.g. the acceptance-gate fix in `default_api_boundary_behavior.rs`), make the branch a strict superset and take the branch's version at merge.

## Default-Boundary Acceptance Gate (default_api_boundary_behavior.rs)

- This is the authoritative locking suite proving the default rs build has no `.NET` runtime. Run the FULL file (`cargo test -p easydict_app --test default_api_boundary_behavior`) when auditing completeness — the `rust-only-boundary` close-out lane runs only 2 of its tests, so regressions in the other ~25 slip past local close-out and only surface in CI (`.github/workflows/ci.yml` runs the full file unfiltered). Treat "rust-only-boundary green" as necessary-but-not-sufficient.
- Windows CRLF pitfall: with `core.autocrlf=true` and no `.gitattributes`, source/manifest files are CRLF on disk, but several of this file's tests `include_str!` the bytes and assert exact `\n`-anchored substrings (e.g. `"\n[features]\ndefault = []\n"`, multi-line `fn` signature splits). Those fail locally with confusing messages like "easydict_app default Cargo features must stay empty" even though the manifest IS correct — it just has CRLF. Do NOT read this as a `.NET` regression. Fix is the `lf()` helper added 2026-06-16 that normalizes `\r\n`→`\n` before matching; apply it to any new `include_str!`/`read_to_string` content that is matched with `\n`-anchored patterns. (`grep`/`od` can mislead about line endings — piping through `awk`/`grep` rewrites them; verify raw bytes with `perl -0777` / hexdump of the unfiltered file.)
- The CLI is intentionally thin: `bin/easydict_cli.rs` must NOT construct the Foundry runtime controller or call `auto_foundry_local_native_probe_request` / `auto_openvino_native_fallback_request`; it delegates LocalAI/Foundry route ownership to the shared app-dir helpers (which use the lib-owned factory internally). Commit 87a3caac removed that dead CLI auto-probe; `default_app_uses_lib_owned_foundry_runtime_controller_factory` was left asserting the CLI still references `default_foundry_local_runtime_controller` and was corrected 2026-06-16 to expect only `lib.rs`/`long_document.rs`/`openai_compatible.rs`/`quick_translate.rs`. If you re-add Foundry construction to the CLI, you are re-introducing the dead probe — don't.

## Iteration Closure

- Keep each migration slice tied to a minimal validation matrix before editing: one formatter check, the smallest behavior/unit tests that prove the changed boundary, and a `gstep diff` scope check before checkpointing. Broader suites should be deliberate risk expansion, not the default closing move.
- When parallel UI work is dirty, isolate those files before Rust core tests and restore them immediately after `gstep commit`; do not let test setup/restoration become an implicit extra refactor.
- Use `rs/scripts/Invoke-RsCoreSliceValidation.ps1` for one-command core validation when known parallel UI/parity files are dirty. Keep custom invocations scoped to a single `rustfmt` or targeted `cargo test`, so failures still point at the current slice instead of a bundled ad hoc suite. When the child command has PowerShell-looking flags such as cargo's `-p`, pass it with an argument array splat (`$cmdArgs = @('cargo', 'test', '-p', 'easydict_app', ...); ...ps1 @cmdArgs`); raw `-File ... cargo -p ...` lets PowerShell capture `-p` before the wrapper sees it.
- Prefer `Invoke-RsCoreSliceValidation.ps1 -CloseOut` for routine final validation. It is the short form of `-RunRecommendedProfiles -CheckTrailingWhitespace`, so it selects the same highest-scoring aligned profile(s), runs the same formatter/test matrix, scans the same changed text files for trailing whitespace, and still accepts `-ChangedPath`, `-AllRecommendedProfiles`, `-MaxRecommendedProfiles`, `-DryRun -Json`, and `-GstepCommitMessage`. Keep `-RunRecommendedProfiles` for explicit matrix debugging and `-Profile <name>[,<name>]` for deliberate fixed-lane runs.
- Prefer `-GstepCommitMessage "<message>"` on the final validation wrapper run instead of running `gstep commit` through a second wrapper invocation. The wrapper runs the checkpoint only after validation and trailing-whitespace checks pass, while the parallel UI/parity files are still isolated, then restores the user's dirty files. It is intentionally blocked with `-NoParallelUiIsolation` so a one-pass close-out cannot absorb parallel UI work by accident. For unusual custom child commands with short flags, use a PowerShell argument array splat instead of relying on `--` through `-File`.
- If the isolated `gstep:@` UI/parity baseline is temporarily compile-coupled with an active parallel UI slice, split close-out deliberately: run the real recommended matrix with `-NoParallelUiIsolation` and no checkpoint, then run an isolated wrapper pass with `-CheckTrailingWhitespace`, the exact `-ChangedPath`, and `-GstepCommitMessage` but no profile or custom command. Record both commands in `refactor-progress.md`; do not claim the single checkpointing command passed when validation and checkpoint had to be split.
- Treat `-ChangedPath` on a checkpointing wrapper run as the commit scope contract. Before `gstep commit`, the wrapper rereads `gstep:@ -> worktree` and refuses to checkpoint any unexpected path that appeared during validation; add intended cleanup paths explicitly, but do not broaden `-ChangedPath` just to hide parallel work.
- A checkpointing wrapper run can legitimately end with no dirty paths if a parallel gstep merge already absorbed the scoped slice while validation was running. In that case the wrapper should skip the post-validation checkpoint with a clear message; do not turn a passed validation into a failure because PowerShell rejected an empty array or because `gstep commit` had nothing left to snapshot.
- Start close-out planning with `Invoke-RsCoreSliceValidation.ps1 -PlanCloseOut` when a slice is already dirty, or pass planned files with `-ChangedPath` before editing. It computes the same recommendation as `-RecommendProfiles`, selects every profile tied for the highest recommendation score by default, adds the trailing-whitespace scan, and prints the ready `-CloseOut ...` command without taking the isolation lock or running cargo. Add `-GstepCommitMessage "<message>"` to the plan when you want the final command preview to include checkpointing, then run the printed command for the actual validation/checkpoint pass. Use `-CloseOut -DryRun -Json` to consume the exact shortcut execution plan, `-RunRecommendedProfiles -DryRun` when you specifically want to exercise that mode's dry-run plumbing, `-MaxRecommendedProfiles N` only when you deliberately want to cap the run, and `-AllRecommendedProfiles` when the current slice should run every matched lane.
- Planned `-ChangedPath ... -DryRun` must work on a clean tree before the first edit. When `gstep diff` is empty, the recommender should accept the empty diff and route by the normalized path list instead of failing parameter binding or returning no plan. Keep a black-box self-test that forces an empty diff with identical selectors, so future tooling edits do not only pass because the wrapper/test files are dirty during the test run.
- Add `-CheckTrailingWhitespace` to the final wrapper run when closing a slice. It reuses the current gstep diff or explicit `-ChangedPath`, skips known parallel UI/parity and generated lock drift, ignores binary/model/dictionary assets, and runs the same `rg -n "[ \t]+$"` check inside the wrapper after the aligned profile succeeds. Use `-DryRun -CheckTrailingWhitespace` first when you want to confirm both the profile matrix and the exact files scanned.
- When docs are updated as part of a code slice, include them in the same `-ChangedPath` used for `-RunRecommendedProfiles -AllRecommendedProfiles -CheckTrailingWhitespace`. The recommender treats `migration-list.md`, `refactor-progress.md`, and `experience.md` as profile-exempt for lane selection, but the trailing-whitespace pass still scans them in the same wrapper run.
- When a slice intentionally needs more than one known lane, pass comma-separated profiles in one wrapper call, for example `Invoke-RsCoreSliceValidation.ps1 -Profile foundry-local,rust-only-boundary`. This keeps formatter/test matrices aligned while avoiding repeated UI/parity isolation and repeated setup work. The dry-run should show profile-prefixed step names for multi-profile runs, and identical commands should appear only once.
- Use `Invoke-RsCoreSliceValidation.ps1 -RecommendProfiles` only when you want to inspect all matching lanes before deciding. The recommender ignores known parallel UI/parity files plus profile-exempt docs, then prints each aligned profile, the default selected profiles, the unique step count, the exact default selected validation steps, a combined `-Profile a,b,c` command, shortcut `-CloseOut` close-out/dry-run commands, explicit `-RunRecommendedProfiles ... -CheckTrailingWhitespace` commands, and all-recommended commands for broader risk expansion. Prefer the shortcut close-out command for routine slices; use `-AllRecommendedProfiles` only when the slice needs every matched lane. When the recommendation was computed with `-ChangedPath` or non-default diff selectors, the printed recommender commands must preserve the same selector inputs. If the changed paths are only the validation tooling, only the tooling self-test lane should be recommended.
- For broad shared test files, `-ChangedPath` close-out should use the actual current diff text as well as the path list. Otherwise touching `cli_translate_behavior.rs` or `quick_translate_behavior.rs` can select every profile that owns one test in that file. Keep the wrapper self-test case that proves custom-streaming CLI diffs select `custom-streaming` instead of dragging in Foundry, traditional HTTP, or rust-only-boundary by path alone.
- When a feature slice also edits `Invoke-RsCoreSliceValidation.ps1` or its self-tests, the default `-RunRecommendedProfiles` selection should include the highest-scoring feature lane plus `core-validation-tooling` regardless of which side scores higher. Keep that rule in the wrapper self-test so tooling changes do not require a remembered manual `-Profile feature,core-validation-tooling` close-out and do not suppress the feature validation.
- Use `Invoke-RsCoreSliceValidation.ps1 -RecommendProfiles -Json` when another script or agent needs to inspect the recommendation matrix, and use `-PlanCloseOut -Json`, `-RunRecommendedProfiles -DryRun -Json`, or `-Profile <name> -DryRun -Json` when it needs the exact execution plan for this close-out pass. Recommendation JSON must keep `DefaultSelectedProfiles` and `DefaultSelectedSteps` as arrays even when there are zero or one selected entries; otherwise every consumer has to carry scalar/object compatibility code and close-out automation drifts. The plan/dry-run JSON report carries selected profiles, per-profile raw `ProfileStepCoverage`, the de-duplicated step list/count, trailing-whitespace paths, checkpoint preview, and ready-to-run `Commands` for the current mode plus the fixed selected-profile equivalent, so automation can align tool test cases without scraping human text, recomputing profile expansion, or rebuilding close-out commands. When auditing a just-checkpointed slice, pass the slice selectors explicitly, for example `-DiffFrom gstep:step-339 -DiffTo gstep:step-340 -ChangedPath ...`; the generated `Commands` must preserve both the changed path list and those diff selectors, otherwise replay falls back to `gstep:@ -> worktree` and broad shared test files lose their diff-keyword precision. Keep tests on the wrapper's shared recommendation/report functions and shared dry-run formatter instead of copying selection or output logic into the self-test harness.
- For shared helper or route files, prefer a dry-run with more than one recommended profile before running cargo. `lib\easydict-windows-text-selection` backs clipboard/text insertion, selected-text capture, and mouse hooks, so use `-RunRecommendedProfiles -ChangedPath lib\easydict-windows-text-selection\src\lib.rs -DryRun -AllRecommendedProfiles` when the diff can affect all three lanes. Shared route files such as `quick_translate.rs` and `long_document.rs` now select tied top lanes by default, while broad `state.rs`/`lib.rs` edits fall back to `rust-only-boundary` when no more specific diff keyword is available.
- For repeated close-out lanes, use `Invoke-RsCoreSliceValidation.ps1 -ListProfiles` and then `-Profile <name>` or `-Profile <name>,<name>` for deliberate combined closure. Profiles isolate the parallel UI/parity files once, automatically enable `RUST_TEST_NOCAPTURE`, and run the aligned tests for that lane (`core-validation-tooling`, `desktop-settings`, `settings-credentials`, `settings-runtime-status`, `app-core-catalog`, `app-preview-window`, `asset-downloads`, `builtin-ai-registration`, `openai-compatible`, `custom-streaming`, `traditional-http`, `translation-cache`, `foundry-local`, `openvino-download`, `windows-ai-native`, `windows-ai-prepare`, `browser-support`, `native-bridge`, `startup-activation`, `protocol-facade`, `input-actions`, `tts`, `file-dialog`, `text-selection`, `mouse-selection`, `ocr-diagnostics`, `longdoc-cli`, `longdoc-script`, `longdoc-layout`, `longdoc-export`, `longdoc-formula`, `mdx-native`, `local-dictionary-suggestions`, `pdf-to-images`, `pdf-overlay`, `store-listings`, `encrypt-secret`, `msix-validate`, `icon-generator`, `runtime-guards`, `windows-registry`, `nllb-native`, `rust-helper-build`, `dotnet-runtime-extract`, `msix-runtime-profile`, `rs-portable-release`, or `rust-only-boundary`). Add a new profile and recommendation rule in the same slice when the same boundary needs both app reducer/lifecycle coverage and lower-level Rust contract coverage; one-off exploration should remain a custom command.
- Use `Invoke-RsCoreSliceValidation.ps1 -Profile runtime-guards` for `lib/easydict-runtime-guards` classifier, byte-scanner, and runtime-profile policy changes. That lane runs the crate once in the default Rust-only feature set and once with `retained-dotnet-workers`; reserve `rust-only-boundary` for app/process/release entrypoint changes that must prove the guard is still wired through the default rs surfaces.
- Use `Invoke-RsCoreSliceValidation.ps1 -Profile windows-registry` for `lib/easydict-windows-registry` HKCU helper changes. Keep this lane focused on the helper's default/named `REG_SZ`, value delete, key delete, and tree-delete semantics; use `desktop-settings` or `browser-support` when the slice changes app shell/protocol/startup or Native Messaging registrar behavior on top of the helper.
- Use `Invoke-RsCoreSliceValidation.ps1 -Profile nllb-native` for `lib/easydict-nllb` manifest, FLORES/BCP-47 mapping, tokenizer, streaming delta, translator, or ORT/OpenVINO helper changes. Keep app route/download/status changes in `openvino-download`, `windows-ai-native`, or the relevant Quick Translate/LongDoc lane, so routine helper work does not pay for app-level route matrices.
- Use `Invoke-RsCoreSliceValidation.ps1 -Profile pdf-overlay` for `lib/easydict-pdf-overlay` harumi wrapper changes. Keep LongDoc export routing, `pdf_native_export.rs`, and overlay block adapter changes in `longdoc-export`, so helper validation stays a quick standalone crate pass.
- Keep `core-validation-tooling` as the contract for close-out tooling changes. It should fail when a validation profile lacks a matching recommendation rule, when a recommendation rule has no profile, when the full profile dry-run wiring no longer expands in one wrapper pass, when tied top recommendations stop expanding together, when shared app fallback paths stop recommending `rust-only-boundary`, when human/JSON recommendation output drifts from the default selected validation plan, when stable JSON array shape breaks for single/empty recommendations, when `-PlanCloseOut` stops producing a copyable no-execution plan, when `-CloseOut` stops matching the recommended profile plan plus trailing-whitespace scan, or when trailing-whitespace/checkpoint close-out switches stop composing with recommended profiles. Keep most assertions in-process by loading the wrapper definition section and directly calling the same recommendation/dry-run functions; reserve black-box wrapper invocations for CLI plumbing, JSON shape contracts, plan output, and argument-error smoke tests, otherwise the self-test cost grows linearly with every new lane.
- For shared files such as `long_document.rs`, put boundary-specific diff examples into the tooling self-test when a manual close-out profile was needed. Cache/quality-report/result-sidecar diffs should default to `longdoc-export`, so future iterations do not pay for the layout matrix just because the runner file also hosts layout plumbing.
- When a second close-out profile catches a regression that belongs to the first recommended lane, move that focused test into the lane before checkpointing. In particular, Foundry Local route work should cover the packaged Auto LocalAI stale app-dir boundary inside `foundry-local`, so `-RunRecommendedProfiles` catches stale `.NET` worker probing and bad Foundry fake-helper fixtures in the first pass.
- Before checkpointing a slice that touches cross-surface routing, process launch, packaged app-dir handling, or broad default CLI/LongDoc behavior, run `-Profile rust-only-boundary` instead of hand-picking no-runtime smoke tests. It is intentionally smaller than the full release gate but covers the common ways `.NET` runtime probing sneaks back into default rs paths; keep its dry-run self-test broad enough to include runtime policy, source/process scans, CLI, GUI stale app-dir, and LongDoc inherited-env boundaries. For direct LongDoc CLI parser/entrypoint work, prefer `-Profile longdoc-cli` first and append `rust-only-boundary` only when the slice also changes shared runtime policy or packaging.
- Do not intentionally run multiple core-slice wrapper invocations in parallel. The wrapper now serializes isolation with a named mutex, but parallel cargo/rustfmt checks still waste time and make outputs harder to read; run formatter and cargo profile/check commands sequentially. If `rs/Cargo.lock` is dirty only because a known parallel UI/parity `Cargo.toml` is dirty, the wrapper temporarily isolates that lockfile too; if a core slice intentionally changes dependencies, keep that manifest/lockfile in the slice and do not rely on UI isolation to hide it. Direct `cargo test --manifest-path` runs against standalone helper crates can generate crate-local `Cargo.lock` files; list those as known generated lock drift in the wrapper so recommendations and checkpoints are not polluted. Generated lock drift that did not exist before the run must be deleted before the checkpoint scope guard, not only in the final cleanup block, because `gstep commit` snapshots the current worktree.
- If a known parallel UI/parity file becomes dirty while a long validation profile is already running, isolate it again immediately before the checkpoint scope guard using a fresh materialization of the current `gstep:@`. Re-isolate paths even when they were already swapped out at the start of the wrapper run, but keep the original backup so the user's in-flight UI edits are restored afterward. This keeps the final `gstep commit` scoped to the requested `-ChangedPath` without forcing a full second validation pass, while still failing on any unexpected non-parallel core path.
- Keep WinFluent platform/tray files such as `win_fluent/src/platform.rs` and `win_fluent_platform_win/src/lib.rs` in the parallel UI/platform isolation list when those slices are owned elsewhere. Do not add broad core app files such as `rs/crates/easydict_app/src/lib.rs` or `quick_translate_behavior.rs` to that list; if they are dirty in parallel, temporarily materialize only those paths from `gstep:@` around the validation/checkpoint run and restore them afterward.
- Do not pass `.ps1` files to `rustfmt`; it treats PowerShell text as Rust and reports noisy backslash escape errors. Validation wrapper self-tests are executed with PowerShell, while the wrapper's profile `rustfmt` steps should list only Rust source/test files.
- When a focused test mutates process-global state such as `LOCALAPPDATA`, put the neighboring behavior tests that exercise the same default path under the same environment lock. Otherwise Rust's parallel test runner can turn a correct diagnostic test into a flaky unrelated failure, costing an extra close-out loop.
- On Windows, a Rust source file can briefly reject overwrite with "user mapped section" while an editor, rust-analyzer, or test process has it memory-mapped. Wrapper cleanup must retry file copies and keep restoring later files even if one restore attempt fails; otherwise a successful `gstep commit` can leave parallel UI/parity backups half-restored. If this ever happens manually, restore the missing files from the wrapper's temp `backup` directory before continuing.

## Clipboard/TextSelection Diagnostics

- Backend errors should not be collapsed into "no text" or timeout if the distinction affects suppression or diagnostics. Preserve backend errors in result enums and only feed confirmed no-text outcomes into suppression.
- Preserving backend errors in a reducer is only half the work. The default app task should expose a Result-shaped completion so UIA/clipboard capture failures can reach settings diagnostics; keep normal fallback success quiet.
- Explicit clipboard read/write task errors should return completion messages; avoid `.ok().flatten()` and `let _ = ...` when errors should be observable.
- Clipboard monitor loops need a bounded diagnostic path. Silently continuing on backend errors hides locked/unavailable clipboard failures, but reporting every poll can spam the UI; emit only distinct errors and clear the monitor-specific message after a successful read.
- The same rule applies to text insertion side effects. `Task::perform(async { let _ = ... }, |_| Noop)` makes Replace/capture failures invisible; return a typed completion message and let success clear only that subsystem's error prefix.
- Settings persistence is also a side-effect future. Do not use `let _ = settings_storage::save_settings_file(...)`; return a typed completion message so write/permission failures are visible, and let success clear only the settings-save error prefix.
- Desktop shell/integration futures should prefix settings diagnostics (`Desktop shell failed: ...`, `Desktop integration failed: ...`) and clear only that prefix on success. Raw error strings make later subsystem-specific recovery checks ambiguous.
- Background registration/writeback futures need the same treatment even when they are startup helpers. Built-in AI device-registration backend errors should be visible, while a later valid token clears only the Built-in AI registration prefix and keeps unrelated subsystem errors intact.
- WindowsAI/Phi prepare can report a semantic failure as `Ok(WindowsAiStatus { state: Failed, ... })`, not only as `Err(...)`. Treat both as app-visible diagnostics, and let Ready/NeedsPreparation/NotCompatible clear only the Phi prepare prefix so unrelated settings errors survive.
- For WindowsAI/Phi route work, prefer `Invoke-RsCoreSliceValidation.ps1 -Profile windows-ai-native` over the older prepare-only lane. The native lane keeps the lower-level `easydict-windows-ai` contract, app prepare reducer, Quick Translate route decisions/client streaming, CLI explicit WindowsAI/no-worker boundary, and LongDoc native route matrix together in one isolation pass.
- Close LocalAI setup/provider slices with their focused profiles: `builtin-ai-registration` for device-registration writeback, `foundry-local` for Rust-owned Foundry prepare and Auto route diagnostics, and `openvino-download` for NLLB/OpenVINO asset download state. These profiles are faster than hand-picking app reducer, OpenAI-compatible, CLI, and LongDoc filters, and they keep the no-worker/stale app-dir checks attached to the route that can regress.
- Auto provider fallback should distinguish a non-ready provider from a broken provider. Foundry Local `NotInstalled` / no endpoint can remain `Ok(None)` so Auto may continue OpenVINO, but `prepare_foundry_local_service(...)` backend errors must stay `Err(...)`; do not hide them behind `.ok()?`, or Quick/LongDoc will replace actionable native diagnostics with generic worker-required fallback text.
- Keep CLI LocalAI route ownership in the shared app-dir helpers. The default `easydict_cli` binary should not call the compatibility `auto_foundry_local_native_probe_request(...) -> Option<_>` helper or keep standalone OpenVINO/Foundry fallback blocks; those are easy to audit as possible error-swallowing paths even when unreachable.
- In Foundry CLI tests, do not use `.cmd` / `.bat` / script files as fake native helpers. The runtime guard is supposed to reject those before spawn. Use injected controllers for backend-error tests, or a missing plain `.exe` name when the contract is the Auto `NotInstalled` / non-ready fallback. If the test needs Auto to reach the Foundry branch deterministically, also set `EASYDICT_WINDOWS_AI_DISABLE_WINRT=1`.
- Screen capture has the same cancellation-vs-backend-failure trap. Keep native OCR capture on a Result-shaped task and reserve `OcrCaptureCancelled` for user cancellation; mapping `Err(...)` to `None` makes GDI/temp-file/native-call failures look like an intentional cancel.
- OCR window snapshots are optional for double-click auto-detection, but their backend errors still matter. Preserve `EnumWindows` failures as diagnostics while keeping manual drag capture available; do not convert them to an empty detector with no clue.
- File dialogs need the same split: `Ok(None)` is user cancellation, `Err(...)` is backend failure. Keep default dialog tasks on Result-shaped completions so COM/thread failures do not look like the user simply dismissed the dialog.
